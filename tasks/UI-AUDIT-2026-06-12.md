# UI/UX AUDIT — 2026-06-12 — Seven Findings
Method: screenshot evidence (dashboard, ORDERS, SMS thread) + targeted code verification. Each finding code-confirmed before listing.

## 1 — Literal code artifacts rendering on the dashboard (VERIFIED)
DashboardTab.jsx:301 renders `TODAY\'S PACE` — backslash visible to the user (screenshot-confirmed). Line 359 has two more: `{TODAY_KEY}` renders a FUNCTION (not `TODAY_KEY()` — React drops it/warns) and `· ${weeklyDialGoal}` puts a stray `$` before the number ("$710 dials/wk"). Your command center has typos in its paint. Fix: 3 one-liners.

## 2 — The dashboard tells a false story every day
"98 BEHIND" pace, "0% contact rate", 2/100 dials — all Funnel-blind. Production lives in Funnel (locked doctrine); Metka only sees Metka work. So the first screen every morning announces failure that isn't real. A dashboard that lies gets ignored — and then it can't warn you of real problems. Fix: rescope the pace/KPI band to the Metka machine (M2/M3 reactivation dials, audits held, sync health, awaiting-reply count) OR add the one-row end-of-day Funnel aggregate and make the numbers whole. This is the highest-value UI change on the board.

## 3 — Lead cards show the STALE phase label, queues sort by the real one
TodaysBlock.jsx:265 (`lead.phase || bucket fallback`) and DialView.jsx:172 (`PHASE_DEFS[open.phase]`) display the stored label, while ordering uses calendar-derived effectivePhase (v3.44). Post-rebase, a card can say "P1" while the engine ranks it M2 — the UI contradicts the doctrine engine on the exact field the whole system runs on. Fix: render effectivePhase(lead) everywhere a chip shows; show "P1 · Day 4" while at it (already-backlogged dialer indicator).

## 4 — Block header announces times that don't exist
TodaysBlock.jsx:196-197: hardcoded "Morning Block 0800–0930 / Afternoon Block 1230–1430" — contradicts the real SESSIONS table (Mon 9:00–10:30, Wed Late 6:30–8:00, Sat double...). The screen literally misstates your own schedule doctrine. Fix: label from getActiveSession()/getNextSession() — "Wednesday Late · 6:30–8:00 · 40 cap" with a countdown when between sessions.

## 5 — Today's Block has no nav entry
The doctrine's PRIMARY operational screen is reachable only via dashboard session tiles. WORK group lists DIAL/SMS/CB/APPTS — but not TODAY. The thing you built the whole machine to feed is a side door. Fix: add `{ id:'today', icon:'⚡', label:'TODAY' }` first in WORK group.

## 6 — alert()/confirm() blocks the dial flow
"✅ Session complete!" alert interrupts auto-advance; SMS failures alert(); delete uses window.confirm. During a 40-lead block, every modal steals keyboard focus and rhythm. Fix: inline toast strip (one small component) for session-complete/SMS results; styled confirm for delete. Lowest urgency, biggest feel-improvement.

## 7 — Speed-readability on the work screens
The screens used at dialing pace run on 8–9px badges, 10px labels, var(--t4) low-contrast text, and small tap targets ("tap to start" implies touch). Tier badges T1/T2 at 8px are unreadable at speed; opted-out ⛔ must never be squint-sized. Fix: minimum 11px on TodaysBlock/DialView/MessagesView interactive text, bump tier/optout badges to 10-11px bold, 44px touch minimums on session tiles + disposition buttons.

## Runners-up (not in the 7): ORDERS row labels are machine-speak ("UNKNOWN DLHA 2026-W23" → "DLHA · week of May 25"); smsUnread only clears from MessagesView (not ContactDetail/DialView threads); CHARGEBACK ACTIVE banner is permanent → alarm fatigue (show until acknowledged weekly); empty SMS thread wastes 80% viewport.
