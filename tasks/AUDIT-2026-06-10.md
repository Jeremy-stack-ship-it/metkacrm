## F1 — HIGH — lastContact UTC/local date split (live operational bug)
Writers disagree on format:
- leads.js:146 (addNote), leads.js:177 (logDial) → `toISOString().split("T")[0]` = **UTC date**
- App.jsx:831 (handleDisposition), App.jsx:869 (handleTodayDispose) → `toLocaleDateString('en-CA')` = **local date**
All readers compare against local: phaseEngine.js:151 (isDueToday), DialQueuePanel.jsx:49, App.jsx:885-888.
Oklahoma = UTC-5/-6. Any dial/note logged after 6-7 PM local stamps TOMORROW's date:
1. Tonight: "contacted today" suppression fails — lead can resurface same evening.
2. Tomorrow: lead is suppressed ALL DAY as "contacted today" when it wasn't.
**Wednesday Late session (6:30–8:00 PM) hits this window every single week.** Every logDial in that session corrupts the next day's queue.
Fix: one shared `localDateKey()` helper; replace both leads.js call sites.

## F2 — HIGH (blocks deployment) — RLS is decorative; auth is cosmetic
rls-setup.sql grants anon USING(true)/WITH CHECK(true) on SELECT/INSERT/UPDATE/DELETE for leads AND activity. The anon key ships in the JS bundle. LoginGate is client-side only (VITE_CRM_PASSWORD also compiled into bundle, visible in source) and gates the UI, not the database.
Today: acceptable ONLY because the CRM is localhost-only. The moment this deploys to app.metkasolutions.com, anyone who views source has full read/write/delete on 2,440 families' PII (names, phones, ages, loan amounts, health notes). That is an E&O / state DOI exposure, not just a tech issue.
Fix before any deploy: Supabase Auth single-user + `auth.role() = 'authenticated'` policies minimum; rotate anon key after.

## F3 — MEDIUM — Two disposition paths, divergent outcomes
App.jsx handleDisposition (dial view, line 719) vs handleTodayDispose (TodaysBlock, line 860). TodaysBlock path is missing: 24-hr no_show auto-callback, noShowSmsQueued flag, 96-hr follow_up_needed callback, disposition notes, direct_vm counter/bucket-advance, direct_vm stage mapping. Same disposition button → different data depending on which screen you're on. no_show dispositioned from TodaysBlock silently loses its recovery-call automation.
Fix: extract one buildDispositionPatch(lead, dispId) used by both.

## F4 — MEDIUM — Phase label never advances with time
phase is set once (backfillLead/phaseFromBucket) or by disposition (no_show→P1, no_sale→P3, terminal→EXIT). Nothing moves P1→P2→P3 as days pass. getPhasePriority ranks on the stale label, so a day-50 lead dialing through its p3_* slots still scores 80 ("P1") and outranks true P2/P3 behavior. Priority queue ordering (no_sale → no_show → P1 → P2 → P3 → M2) is therefore distorted for every lead older than 14 days that was never dispositioned into a new phase.
Fix: derive effective phase from phase_start age (or earliest unconsumed sched col) inside getPhasePriority.

## F5 — MEDIUM-LOW — masterQueueSort tiebreaker inverts urgency
phaseEngine.js:373 `bTs - aTs` = due-LATER-first within same phase. Most-overdue leads sort last. If intentional staleness deprioritization, document it; if not, flip to aTs - bTs.

## F6 — MEDIUM-LOW — deleteLead/addLead still use stale `leads` closure
leads.js:208-250 read the memoized closure array and call saveLeads(next) — the pattern v3.12 eliminated from upd(). Deleting/adding mid-dial-burst can persist a pre-disposition snapshot of the entire array. Low frequency, high blast radius.
Fix: route both through setLeads(prev=>) like upd().

## F7 — LOW — Startup migrations don't bump _ts or push to Supabase
spreadOverdueLeads/normalizePhaseSchedule mutate next_dial locally, persist to localStorage, never upsert. Remote keeps stale next_dial. Harmless single-device (equal-_ts keeps local), diverges the day a second device appears.

## F8 — LOW (retro) — v3.38 phone dedup may have removed spouses
Dedup keyed on bare phone digits, newest _ts wins. Mortgage-protection households routinely share one number — second spouse records would have been tombstoned. One-time migration already ran. Action: skim LS_DELETED_IDS tombstones from ~v3.38 date for recoverable names.

## F9 — LOW — addNote stamps lastContact for every note type
leads.js:146 — writing a plain journal note marks the lead "contacted today" and pulls it from today's queue (isDueToday suppression). Notes are not contacts.

## F10 — LOW — sbBeaconFlush 64KB keepalive cap
Full lead objects include entire notes arrays. A few dozen dirty leads with long histories can exceed the browser keepalive limit → silent drop (fetch rejection is async, the try/catch never sees it). Consider flushing promoted columns + last note only.
